# Rumour

Anonymous, room-code based chat for Flutter. Users join or create rooms with a code, chat in real time, and are identified by a generated profile (no accounts).

APK: https://drive.google.com/file/d/1LHiJvtZnTTiMaHSXaju0gIYtQNStLinC/view?usp=drive_link
Demo walkthrough: https://drive.google.com/file/d/11ziuphlWEzZPmzHEXJG70tf2wilm2oFW/view?usp=drive_link

## Stack

- **Flutter** (Dart ^3.8), **Material 3** with light/dark theme and clamped text scaling
- **Firebase**: Core + **Cloud Firestore** (rooms, participants, messages; offline persistence enabled at startup)
- **flutter_bloc** + **equatable** for presentation state
- **HTTP**: RandomUser API for anonymous display name and avatar; local **UUID** for user id; device id for participant documents

## Features

### Onboarding and navigation

- **Splash**: Short branded intro, then automatic navigation to the join screen.
- **Join room**: Enter a **6-character** room code (**A–Z** and **0–9** only). Input is normalized (uppercase, filtered) and validated before any network call. From this screen you can open **Create room** or submit a code to enter chat.
- **Create room**: Picks a **unique display name** (e.g. `Room #NNNN` style) with Firestore collision retries for the preview. On confirm, the app generates a **unique room code** (again with retries if the code is taken), creates the room document, and adds the creator as a participant.

### Identity (no accounts)

- Each device uses a stable **device id** for the participant document key (`participants/{deviceId}` under the room).
- **User id** is a local **UUID**; **display name** and **avatar** come from the **RandomUser** HTTP API when possible, with a **safe fallback** if the request fails so chat still works offline or when the API is down.
- **Join flow**: If this device already has a participant row for the room, the app reuses it; otherwise it creates a new participant with a fresh anonymous profile.
- **Chat entry**: A **resolve session** step loads or creates the participant record before the message list is fully active, so first-time joins are handled in one place.

### Chat

- **Realtime messages** from Firestore with **pagination**: older messages load when you scroll toward the **top** of the reversed list (infinite-style “load more”).
- **Date separators** (e.g. Today, Yesterday, or a calendar date) group the timeline.
- **Bubbles** distinguish **your** messages from others using the resolved participant **user id**.
- **Live member count** for the room (streamed from Firestore).
- **First-time in chat**: Optional acknowledgment UI before full interaction (e.g. after joining as a new participant).
- **Theme**: Light/dark toggle available from the chat UI; app-wide **theme mode** is preserved for the session.

### Platform and UX

- **Mobile**: Standard full-screen layout.
- **Web**: Content is **width-constrained** with top padding so the app reads well on large monitors; **text scaling** is clamped for readability.

## Project layout

| Area | Role |
|------|------|
| `lib/app/` | Root `MaterialApp`, theme mode |
| `lib/core/` | Routing, theme, constants, shared services (Firebase bootstrap, Firestore, API, device/user identity), utilities |
| `lib/features/` | **splash**, **room** (create/join, participants), **chat** (session + messages) |
| `lib/shared/` | Cross-feature widgets |

Each feature follows **domain** (entities, repository contracts, use cases, validators) → **data** (Firestore repos, models, mappers) → **presentation** (BLoC, screens).

## Run

1. Install [Flutter](https://docs.flutter.dev/get-started/install) and run `flutter pub get`.
2. Add a Firebase project and place `firebase_options.dart` / platform configs as generated by FlutterFire (`flutterfire configure`).
3. `flutter run` (or build APK/IPA as needed).

## Notes

- Route names live in `lib/core/constants/app_route_paths.dart`; generation is in `lib/core/routing/app_router.dart`.
- Repository calls return a `RepositoryResult` type; data errors use `DataLayerException` where applicable.
